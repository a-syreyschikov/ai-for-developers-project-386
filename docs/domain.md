# Доменная модель

Проект `calendar` реализует упрощенный сервис записи на звонок по мотивам Cal.com. В MVP нет регистрации, авторизации, личных кабинетов и внешних календарей.

## Глоссарий

- владелец календаря — `Owner`
- тип события — `EventType`
- окно доступности — `AvailabilityWindow`
- слот — `Slot`
- бронирование — `Booking`

## Роли

- `Owner` — единственный заранее заданный профиль владельца календаря. Owner API в MVP не защищен авторизацией.
- Гость — пользователь без аккаунта, который смотрит типы событий, выбирает свободный слот и создает бронирование.

## Сущности

`Owner` содержит UUID, отображаемое имя и IANA time zone. `timeZone` нужен для отображения времени, но все API timestamps передаются в UTC.

`EventType` описывает вид звонка: UUID, название, описание и длительность в минутах. Название уникально после trim и case-insensitive нормализации.

`AvailabilityWindow` описывает конкретный UTC-интервал, опубликованный владельцем как доступное время. Окна доступности не являются слотами; слоты вычисляются из окон для выбранного типа события.

`Slot` — вычисляемый свободный интервал для конкретного `EventType`. Слот не хранится как ресурс, не имеет `id` и не раскрывает `availabilityWindowId`.

`Booking` — запись гостя на конкретный слот. Бронирование хранит summary типа события для отображения списков без дополнительных запросов.

## Время

- Все timestamps передаются в UTC ISO 8601 с суффиксом `Z`.
- Допускается только минутная точность: секунды и миллисекунды должны быть нулевыми.
- Все интервалы полуоткрытые: `[startsAt, endsAt)`.
- Источник `now` — серверное UTC-время.
- Окно записи гостя: `now < startsAt < now + 14 days`.
- Правая граница окна записи не включается.
- Новые окна доступности должны иметь `startsAt > now`.
- Границы окон доступности должны быть выровнены по 15-минутной сетке.

## Доступность и слоты

- Владелец создает, просматривает и удаляет окна доступности.
- Владелец может публиковать окна дальше чем на 14 дней вперед.
- Пересекающиеся окна доступности запрещены.
- Окно доступности может пересекаться с существующими бронированиями.
- Шаг генерации слотов равен `durationMinutes` выбранного типа события.
- Слоты сначала строятся сеткой от `AvailabilityWindow.startsAt`, затем фильтруются по scheduled бронированиям.
- Слот показывается только если полностью помещается в окно доступности.
- Остаток окна меньше длительности события игнорируется.

## Бронирования

- Гость отправляет `eventTypeId`, `startsAt`, `guestName`, `guestEmail`.
- Backend вычисляет `endsAt` по длительности типа события.
- Нельзя создать два пересекающихся scheduled бронирования даже для разных типов событий.
- Отмененные бронирования не блокируют слот.
- Владелец может отменить будущую scheduled запись.
- Повторная отмена идемпотентна.
- Отменять прошедшее или уже начавшееся бронирование нельзя.
- Список upcoming содержит только `scheduled` бронирования с `startsAt > now`, отсортированные по `startsAt ASC`.

## Валидация

- Все resource `id` генерируются сервером как UUID.
- `Owner.displayName`: 1..120 символов.
- `EventType.title`: 1..120 символов.
- `EventType.description`: обязательная строка 0..1000 символов.
- `EventType.durationMinutes`: 15..240 минут, кратно 15.
- `guestName`: 1..120 символов.
- `guestEmail`: валидный email 1..254 символа.
- Строки перед сохранением trim-ятся.
- Email дополнительно нормализуется в lower-case.

## Ошибки

API возвращает RFC 7807 `ProblemDetails` с типизированным `code`.

- `400 Bad Request` — ошибки структуры и базовой валидации.
- `404 Not Found` — ресурс не найден.
- `409 Conflict` — конфликт текущего состояния.
- `422 Unprocessable Entity` — доменное правило нарушено.
